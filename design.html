SRE Design

00Architecture & Component Map

A synchronous, stateless service (each request is independent — no session memory) on the checkout critical path (every cart-pricing request goes through us). Target: p99 < 150 ms. If we're slow or down, customers can't buy — that's the rule the rest of the design follows.

Hover any box to see what it does and why · scroll sideways on small screens
Checkout Service
the app that calls us
WHY Upstream Caller Calls discount-service synchronously on every cart-pricing request. Checkout-blocking dependency — if we're slow or down, customers can't buy.
  • Deadline propagation (passes a "how long you have" header) — we reject early if we can't meet it
  • Latency > 150 ms = effectively unavailable (upstream timeout fires)
  • Retries on transient failures → drives our idempotency requirement (same input, same answer, every time)
discount-service
Python · 6 to 100 copies running
WHY Core Service FastAPI/uvicorn async service. Stateless (no session memory between requests), so the HPA can add or remove pods freely.
  • minReplicas = 6 · 3 AZ × 2 pods/AZ — survives a full AZ failure
  • maxReplicas = 100 · ~1.5× peak demand of ~67 pods, headroom for assumption-error
  • HPA at 60% CPU — Python GC (garbage collection — periodic memory cleanup) pauses spike individual pods to 95%+; 60% leaves headroom before request queueing kicks in
  • podAntiAffinity (no two pods on same node) — eliminates shared-fate failures
/calculate
price the cart
WHY Hot Path Endpoint The 95% case — every cart load hits this. Pure read, naturally idempotent (same input always returns the same answer, no matter how many times you call it).
  • No side effects — safe to retry without idempotency keys
  • Determinism enforced — no datetime.now() inside calculation, except for explicit voucher expiry checks
  • Cache hit (~85%): ~12 ms. Miss path: 50 ms DB budget
/validate_code
check a coupon
WHY Voucher Path Different from /calculate — vouchers can be revoked mid-session, single-use codes have state, fraud checks have side effects.
  • 30s cache TTL (time-to-live — how long a cached entry stays valid) vs. 5 min for rules — bounds the revocation window
  • Idempotency key required — retried request must not double-decrement single-use redemption counters
  • Voucher codes SHA-256 hashed (one-way scrambled) in logs — log access ≠ replay surface
Redis (rules)
held for 5 minutes
WHY Primary Cache Without this layer, 20K RPS peak isn't serveable — Postgres alone can't handle it. The cache is what makes peak survivable.
  • Cache key: brand+category+bank — caches at rule-set level (shared across carts), not per-cart
  • volatile-lru eviction (only items with a TTL get dropped under memory pressure), not allkeys-lru
  • Hot rule keys pinned without TTL — bounds the eviction blast radius (limits how much breaks if eviction kicks in)
  • Hit ratio > 85% steady; drop below 80% leads p99 latency by 5–10 min
Redis (vouchers)
held for 30 seconds
WHY Voucher Cache Short TTL on purpose. Codes can be revoked (fraud, expiry, single-use redemption). A 5-min window where a revoked code still validates is a real abuse surface.
  • 30s caps the abuse window while still cutting most DB load on hot codes
  • Separate keyspace from rules — different eviction behavior, different failure mode
PgBouncer
shares database connections
WHY Connection Multiplexer Mitigates the first peak bottleneck. 100 pods × 15 connections = 1,500 — close to Aurora's ~1,700 max_connections ceiling. PgBouncer gives us headroom.
  • Transaction-pooling mode — multiplexes per transaction (reuses connection between requests, not for the whole session)
  • ~200 server-side connections, multiplexes thousands of client-side connections through them
  • v1.5 hardening — omitted from simplified diagram for readability; mandatory before first peak-sale event
Aurora PostgreSQL
main + read-only copy
WHY Source of Truth Aurora over vanilla RDS — failover 15–30s vs. 60–120s (failover = automatic switch to a healthy replica when the primary dies). On checkout, that gap is material.
  • Read replica (read-only copy that stays in sync) at peak — splits 3K q/s rule lookups across two endpoints
  • Dataset tiny (~5 MB) — IOPS is never the bottleneck; connection count is
  • Failure mode #1: sequential scan (DB reads every row instead of using an index) on a new bank-offer column. CI runs EXPLAIN ANALYZE on every rule migration to catch it
  • All migrations backward-compatible with N-1 code (works with both old and new app versions) — keeps rollback < 5 min
Datadog
logs, metrics, alerts
WHY Observability Unified UI for logs, metrics, and distributed traces. At 3 AM, jumping between three tools to correlate is friction we don't need.
  • RED metrics (Rate, Errors, Duration) + USE (Utilization, Saturation, Errors)
  • discount_applied_amount_cents histogram — catches silent miscalculation bugs latency metrics never will
  • Cardinality discipline — number of unique label values stays bounded; voucher_code goes in logs only, never as a metric label
  • Trade-off: vendor cost vs. unified UI — chosen explicitly
Secrets Manager
stores passwords safely
WHY Secrets & Supply Chain DB credentials and Redis auth tokens injected at pod startup via the Secrets Store CSI driver (Kubernetes plugin that mounts secrets from cloud secret managers as files). Never baked into images.
  • No secrets in Docker build args, env files, or git history
  • git-secrets pre-commit hook + CI scan block accidental check-ins
  • Image signing via AWS Signer; OPA admission policy (cluster gatekeeper rule) enforces signed images only
Auto-scaler
adds pods when busy
WHY Elasticity HPA (Horizontal Pod Autoscaler — adds/removes pod replicas based on CPU) + Cluster Autoscaler (adds/removes whole VMs). Asymmetric stabilization (different speeds for up vs. down) on purpose.
  • scaleUp stab. 60s — sale traffic ramps in seconds; slow scale-up = dropped requests
  • scaleDown stab. 300s — prevents post-spike thrash (rapid up-down oscillation)
  • Pre-scale 30 min before known sale events; node provisioning takes 2–3 min

01Observability

Three pillars: logs (what happened to one request, written line-by-line), metrics (numerical trends over many requests, aggregated), and distributed traces (the path one request took across services, with timing per hop). All three feed Datadog so the on-call has one place to look.

Structured Logging (one JSON line per request)

Each request emits one machine-parseable line with: trace_id (correlates the whole request across services), request_id, duration_ms, cache_hit, discount_types_evaluated, customer_segment, and error fields when applicable.

Cache hit, success

{
  "level": "INFO",
  "event": "calculate_cart_discounts.success",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "duration_ms": 12,
  "cache_hit": true,
  "discount_types_evaluated": ["brand", "category"],
  "customer_segment": "loyalty_gold",
  "original_price_cents": 8500,
  "final_price_cents": 6800
}

Database is slow

{
  "level": "WARN",
  "event": "calculate_cart_discounts.slow",
  "duration_ms": 134,
  "cache_hit": false,
  "db_query_ms": 121,
  "warning": "db_latency_approaching_slo_budget"
}

What we never log: raw voucher codes (we hash them), full customer details, full cart contents.

Metrics — RED + USE + Business

RED (Rate, Errors, Duration — request-level health) and USE (Utilization, Saturation, Errors — resource-level health) are the standard SRE base. We add business-correctness metrics on top.

MetricTypeWhy we want it
discount_request_totalCounterRED: rate, by endpoint and outcome
discount_request_duration_secondsHistogramp99 latency SLO (target percentile) is computed from this
discount_cache_hit_ratioGaugeLeading indicator — drops precede latency by 5–10 min
discount_db_pool_utilizationGaugeUSE saturation — first peak bottleneck
discount_applied_amount_centsHistogramBusiness correctness — catches silent miscalculation regressions
discount_zero_applied_totalCounterSpike → voucher system or rule misconfiguration
WHY
Why discount_applied_amount_cents is the most important metric here

If a code change accidentally zeroes out all bank discounts, error rate is fine and latency is fine — but every customer is paying full price. Watching the average discount given catches this in minutes. Without it, you find out from customer complaints hours later.

CARDINALITY
High-cardinality labels are forbidden on metrics

Cardinality = number of unique label-value combinations. Each unique value becomes a separate stored time-series. voucher_code, customer_id, request_id — anything with millions of unique values — would explode the cost (per-series billing in Datadog) or OOM the scrape target (in Prometheus). They live in logs and traces only.

Distributed Tracing (OpenTelemetry → Datadog APM)

A trace is the path one request took, broken into spans (each unit of work — one Redis call, one DB query). Spans are nested to show the call tree.

[HTTP: calculate_cart_discounts]            # total span
  ├─ [redis.get rules_cache_lookup]         key=brand:zara|cat:dresses|bank:hdfc
  ├─ [postgres.query fetch_discount_rules]  # only on miss
  │     db.rows_returned=12
  ├─ [redis.set rules_cache_populate]       ttl=300s
  └─ [app: apply_discounts]
        discount.types_evaluated=["brand","bank"]
        discount.applied_count=2

02SLI / SLO & Error Budget

An SLI (Service Level Indicator) is a precisely-defined measurement (e.g. "fraction of requests under 150 ms"). An SLO (Service Level Objective) is the target for that SLI (e.g. "99.9% over 28 days"). The error budget is the allowed shortfall — when we burn through it, we stop shipping features and fix what's wrong.

SLI Definitions

SLI-1 · Availability

Fraction of requests returning HTTP 2xx AND finishing within 150 ms, over a rolling 28-day window. Combined definition: a 200 ms response is functionally the same as a 5xx error — checkout has already timed out.

SLI-2 · Correctness

Fraction of calculations where final_price ≤ original_price AND applied_discounts is non-empty when at least one rule matched. Validated by a shadow comparison job (runs alongside production traffic, checks expected outputs) against fixtures every 5 minutes.

SLI-3 · Cache hit ratio (leading indicator, not SLO)

5-min rolling cache hit ratio. Tracked but not an SLO target — drops here predict latency SLO breach by 5–10 min.

SLO Targets

SLOTargetWindowWhy this number
Availability99.9%28 daysError budget = 43.8 min/month. Honest for single-region multi-AZ; 99.95% requires multi-region active-active.
Latencyp99 < 150 ms5 minSpec requirement. Owned despite Python GC pauses making it costly.
Correctness99.99%7 daysTighter than availability: wrong price > no price.
WHY
Why correctness is set higher than availability

Asymmetric cost. Overcharging is a legal/trust issue (refunds, lawsuits). Undercharging is silent revenue loss — no customer complains when they pay less. So the bar for "did we calculate right" must be stricter than "did we respond at all."

Error Budget Policy

ConsumedOperational change
< 50%Normal ops. Feature work and deployments proceed.
50%Eng lead notified. New feature deploys need SRE sign-off. Top error sources reviewed.
75%Feature freeze (no new functionality). Reliability fixes only.
100%Full freeze. Incident retrospective required. SLO target reassessed with leadership.

Decision rights: Eng manager + SRE lead at 50/75%. VP Eng at 100%. Correctness breaches bypass the budget — they escalate immediately as incidents. External-event budget burn (AWS AZ outage, upstream dependency failure) is reviewed separately — we don't penalize the team for failures outside our control.

WHY
HPA target at 60% CPU, not the textbook 70%

Python's GC (garbage collection — periodic memory cleanup that pauses the process) creates uneven CPU. At 70% average utilization, individual pods spike into the 90s during GC, async event-loop queueing kicks in, and p99 breaches. 60% gives ~15% headroom before queueing. If load testing shows GC pressure is mild, 65% is defensible.

Top 3 Failure Modes

1 · Sequential scan on discount_rules after a new bank offer

A new offer with applicable_card_types = ARRAY['visa','mastercard','amex'] defeats the GIN index (Postgres index type for array/JSON columns) if it doesn't cover the new pattern. Plan flips to Seq Scan (full table read instead of index lookup). Query p99 jumps from ~5 ms to 200+. db.rows_returned spikes 10×. Connection pool saturates.

Mitigation: CI runs EXPLAIN ANALYZE (Postgres command that shows the actual query plan) against a production-scale staging dataset for any rule-column migration. Read replica for rule queries. If it lands in prod: bump Redis TTL 5m → 30m while CREATE INDEX CONCURRENTLY (creates an index without locking writes) runs.

2 · Redis eviction during Black Friday cascades to Postgres

Default allkeys-lru evicts any key under memory pressure. Cold cache at 20K RPS = 20K queries/sec to a DB sized for 200–400/sec. Pool saturates in seconds — the thundering herd (many requests piling onto a slow dependency at once).

Mitigation: Pre-size Redis (rule set is < 100 MB — eviction implies a leak). Use volatile-lru so only TTL'd keys are evicted; pin critical rule keys without TTL. Circuit breaker (auto-trips when a downstream is failing): if pool utilization > 90% for 10s, serve stale rules instead of queueing more DB connections.

3 · Silent miscalculation for a brand+bank combination

A rule-priority ordering bug applies brand and bank discounts multiplicatively instead of additively for Zara + HDFC. Final price 30% lower than intended. This will not show in error rate or latency metrics.

Mitigation: Correctness shadow job (SLI-2) every 5 min against known fixtures. Canary uses discount_applied_amount_cents distribution — auto-rollback on > 15% shift. Daily finance reconciliation against expected vs. applied totals; > 0.1% deviation pages.

Graceful Degradation

The central tension on the checkout path: fail-open (degrade gracefully and serve a reduced response) vs. fail-closed (refuse the request entirely). Decision: fail open.

  • Redis unreachable → bypass cache, query Postgres directly. Doubles DB load but stays correct.
  • Postgres unreachable + cold cache → return zero discounts (final_price = original_price, message = "Discounts temporarily unavailable"). Customers complete checkout at full price.
  • Postgres slow (> 100 ms) → serve stale cached rules up to 10 min old.
  • Never acceptable: final_price > original_price (legal/regulatory exposure) or substantially under-priced (revenue exposure). Returning the original price is the conservative-safe fallback.

04CI / CD & Change Safety

Pipeline gates from PR to production, with extra checks for discount-logic changes. Standard signals (error rate, p99 latency) miss silent pricing regressions — we layer correctness checks on top.

The pipeline

PR opened
  ├─ formatters · type checks
  ├─ unit tests on discount logic
  ├─ security scan on dependencies + code
  ├─ container scan · blocks high/critical issues
  └─ review · 1 SRE + 1 domain engineer for logic changes

Merge → main
  ├─ integration tests · real Postgres + Redis
  ├─ schema check · don't break checkout's contract
  └─ build, push, deploy to staging

Staging
  ├─ smoke tests
  ├─ load test · 5 min at 2× normal · p99 must stay < 120 ms
  ├─ 50 known carts with exact expected prices
  └─ manual sign-off for logic changes

Production
  ├─ canary 5% · 10 min
  │     auto-rollback if: errors > 0.5% · p99 > 140 ms · discount drift > 15%
  └─ progressive 25 → 50 → 100% in 10-min steps

Discount-Logic Rollout Gates

Three additional gates on top of the standard pipeline:

  1. Fixture suite — 50 known cart/rule combinations with exact pre-computed expected prices, run in staging. Primary correctness gate.
  2. Canary metric — distribution comparison. p25/p50/p75 of discount_applied_amount_cents compared between canary pods and stable. Shift > 15% triggers auto-rollback.
  3. Shadow mode — for major logic changes, run new logic in parallel for 24 hours, return the old answer to users, alert on output divergence. Zero customer impact, full visibility into what would have changed.

Migration policy: all schema changes backward-compatible with N-1 code (works with both the previous and current app version). Additive only — no breaking renames or constraint changes. Renames split into add column → backfill → drop, across separate deploys. This keeps the rollback path always trivial.

Throughput Math

Per-pod throughput estimate: ~300 RPS for Python async (FastAPI/uvicorn) at 20 ms avg latency, accounting for GC, pool contention, and TLS overhead. This is the assumption the entire capacity model rests on — validate by load test on day one.

BOTTLENECK
Postgres connection ceiling, not CPU or IOPS

If the 300 RPS/pod assumption proves optimistic and pod count climbs toward 100, we approach Aurora's max_connections ~1,700 limit before saturating CPU or disk. Mitigation: PgBouncer in transaction-pooling mode — holds ~200 server-side connections, multiplexes thousands of client-side connections through them. Mandatory before peak-sale readiness.

Backpressure & Admission Control

At 20K RPS, queueing causes tail latency collapse: requests pile up faster than they drain, average latency climbs into multi-second territory before timing out, and the system becomes uniformly broken instead of partially serving. Three controls prevent it:

  • Deadline propagation: checkout passes a deadline header (how long it can wait). We reject early with 503 deadline_exceeded rather than start work we can't finish in time.
  • Bounded request queue per worker (max 50 in-flight). Above the cap, fail fast with 503 overloaded. Controlled rejection > cascading timeout.
  • DB pool wait cap: if a request waits > 30 ms for a connection, abort and return zero-discount fallback rather than block the worker.

Idempotency & Retry Safety

Checkout retries on transient failures. The calculation is naturally idempotent (same inputs → same output), but invariants must hold:

  • Pure functions: rule evaluation never depends on wall-clock time except explicit voucher expiry checks (deterministic given the input timestamp)
  • Idempotency key on validate_discount_code — the request-ID short-circuits duplicate redemption-counter increments on retry
  • No side effects in the calculate path. Cache populates are safe to repeat

Cost Optimization — Spot Worker Tier

Spot instances (spare AWS capacity at 60–70% discount, reclaimable with 2-min warning) for non-baseline pods. Run 6 on-demand pods (the reliability minimum); elastic capacity above that floor on Spot. Estimated $200–300/month savings, scaling with peak fleet.

Black Friday exception: 100% on-demand. Spot reclamation rates spike during high-demand windows — exactly when we can least afford a forced reclamation cascade. Pre-purchase Savings Plans for the baseline.

06Runbook · DB Pool Saturated

A real on-call procedure. Optimised for 3 AM cognitive load: ranked likely causes, explicit verification before closing, escalation timeline.

Symptoms

  • PagerDuty: DiscountService_DBPoolSaturation
  • Pool utilization gauge red
  • p99 latency climbing from ~20 ms toward 150 ms
  • Logs show WARNING pool_wait_ms=… or ERROR pool_timeout

Is it the pool, or the database?

discount_db_pool_utilization                            # > 90% confirms pool
discount_db_query_duration_seconds{quantile="0.99"}     # is the DB itself slow?

Pool high + queries fast → connection-count problem. Stay here.
Pool high + queries slow → DB is the cause. Different runbook.

What changed (most likely first)

  1. Traffic spike — check requests/sec
  2. Recent deploy bumped pool size — check kubectl describe deploy
  3. HPA hasn't caught up — check kubectl get hpa
  4. Cache hit rate dropped — doubles DB load
  5. A slow query showed up — holds connections longer

Fix it

# First: check we won't blow the DB connection limit
psql -h $DB_HOST -c "SELECT count(*) FROM pg_stat_activity;"

# Force scale-up
kubectl scale deployment discount-service --replicas=50

# Bad deploy → roll back
kubectl rollout undo deployment/discount-service

# Last resort: shrink per-pod pool
kubectl set env deployment/discount-service DB_POOL_SIZE=5

Confirm recovery

All three must be trending right before closing:

  • Pool utilization < 70%
  • p99 latency < 50 ms
  • Error rate at zero

07Assumptions & Trade-offs

Assumptions

  • Python async (asyncio + FastAPI/uvicorn), not synchronous WSGI. Capacity math changes substantially under sync workers.
  • Aurora PostgreSQL, not vanilla RDS. Failover is faster (15–30s vs. 60–120s) and read replicas are trivial to add.
  • Datadog. Unified UI for traces/metrics/logs at 3 AM matters more than vendor cost. Prometheus + Grafana + Jaeger is a valid alternative.
  • Voucher TTL = 30s because codes can be revoked. 5-min TTL is acceptable if revocation isn't a requirement.
  • Per-pod throughput ≈ 300 RPS at 20 ms avg. Validate by load test before treating as fact — the entire capacity model rests on this number.

Explicit Trade-offs

TRADE-OFF
99.9% availability SLO, not 99.95%

Honest for single-region multi-AZ. 99.95% would require multi-region active-active (both regions serving traffic concurrently with conflict resolution) — significant operational complexity. Aurora failover alone consumes 15–30s of the budget. Promising 99.95% on this architecture would be dishonest.

TRADE-OFF
Fail-open with zero-discount fallback, not fail-closed

When dependencies break, customers complete checkout at full price rather than being blocked. The cost of "no discount" is bounded; the cost of "no checkout" is unbounded (lost transaction + churn). This must be a product decision, not an engineering one.

TRADE-OFF
HPA target 60% CPU, not 70%

More conservative than tutorials. The 10% extra headroom absorbs Python GC pauses (which spike individual pods to 95%+). Validated by load test — raise to 65% if GC pressure proves mild.

TRADE-OFF
PgBouncer omitted from v1 architecture diagram

Adds operational complexity. Diagram simplified for the v1 launch story. Ships in v1.5 — mandatory before first peak-sale event.
